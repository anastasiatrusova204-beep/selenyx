## Принципы — чего не повторять за конкурентами

Из отзывов на Saturn.Love, Numia, Co-Star, Moonly, The Pattern — что раздражает пользователей больше всего:

| Что бесит | У кого проблема | Наше правило |
|---|---|---|
| Платить за каждую функцию поверх подписки | Moonly, Co-Star | Платный контент не дублирует бесплатный |
| Много уведомлений | Все | Максимум 1 сообщение в день |
| Гороскоп «для всех» без персонализации | Все | Всегда учитываем знак пользователя |
| Узнать о платном контенте после регистрации | Saturn.Love | Цены видны до оплаты |
| Ответ ждёшь дольше 3 секунд | Все | Бот отвечает мгновенно |

> Нарушение любого из этих правил = потеря пользователя. Это не предположение — это жалобы №1 в реальных отзывах.

```bash

```

Внутри контейнера:
```bash
# Сколько уникальных пользователей зашло
sqlite3 /data/selenyx.db "SELECT COUNT(DISTINCT user_id) FROM event_log;"

# Что открывали чаще всего
sqlite3 /data/selenyx.db "SELECT event, COUNT(*) FROM event_log GROUP BY event ORDER BY 2 DESC;"

# Кто зашёл больше одного дня (retention)
sqlite3 /data/selenyx.db "SELECT user_id, COUNT(DISTINCT date(ts)) as days FROM event_log GROUP BY user_id HAVING days > 1;"
```

Что ищем: открывают ли бот на 2-й и 3-й день? Если да — продукт работает.

## Шаг 7.5 — Нумерология дня

**Результат:** в «Мой день» появляется число дня (1–9) с коротким текстом — контент меняется каждый день, не требует данных от пользователя.

Что делаем:
- Функция `get_day_number(date)` — складывает все цифры даты до однозначного числа (например, 8+3+2026 = 21 → 3)
- Словарь `NUMEROLOGY_DAY` — 9 текстов, по одному на каждое число
- Показываем в «Мой день» новой строкой: `· 🔢 3 день — ...`

**Второй уровень (с натальной картой, Шаг 11):**
- Число судьбы — из даты рождения пользователя
- Число личного года — число судьбы + текущий год
- Персональный совет на основе числа судьбы

Проверка: каждый день в «Мой день» показывается новое число и текст, совпадает с нумерологическим расчётом для текущей даты.

## Шаг 13 — Видимость и продвижение бота

### Поиск внутри Telegram (Telegram SEO)
Telegram ищет ботов только по **username** и **имени бота** — по описанию не ищет.

Что влияет на нахождение в поиске:
- `@username` — сейчас `@Selenyx_mybot`. Если хочется попадать в поиск по «луна» / «астро» — нужен username с ключевым словом (например `@AstroSelenyx`). Менять только если бот ещё не раскручен.
- **Имя бота** (не username) — тоже участвует в поиске. Можно добавить ключевое слово: `Selenyx — лунный навигатор`.
- **Количество пользователей** — чем больше база, тем выше бот в результатах поиска.

Вывод: поиск Telegram даёт малый трафик на старте. Главный канал роста — прямые ссылки.

### Продвижение (реальные каналы трафика)

## Шаг 10.5 — Совместимость двух знаков
**Результат:** пользователь проверяет совместимость своего знака с другим — делится результатом.

## Принципы контента — чему учит конкуренция

Эти принципы должны выполняться с первого дня, не когда-то потом:

**1. Психологический фрейм (The Pattern)**
Тексты бота — не «что произойдёт», а «как лучше использовать текущую энергию». Слова «произойдёт», «случится», «ждёт» — вон. Слова «поддерживает», «помогает», «самое время для» — в обиход. Это убирает тревогу и добавляет ощущение контроля.

**2. Читается за 30 секунд (Co-Star, Moonly)**
Максимальная длина одного экрана. Если текст не помещается в одно Telegram-сообщение без скролла — нужно сокращать, а не добавлять.

**3. Контент, который хочется переслать (The Pattern)**
Каждое сообщение должно быть ценным само по себе. Тест: хочется ли скинуть скрин подруге? Если нет — текст нужно переписать.

**4. Растущее качество текстов (Co-Star)**
MVP: авторские шаблоны. При базе 500+ пользователей: GPT-генерация на основе астрологических данных. При монетизации: привлечение астролога-соавтора.

## Анализ рисков — что может пойти не так (март 2026)

Критический разбор слабых мест продукта до закрытого теста.

### Бизнес-модель

**Риск 1 — Разовые покупки без повторного дохода**
Потолок с одного пользователя: 2170₽ (690 + 990 + 490). После покупки всего — платить больше не за что. Нет подписки → нет предсказуемого дохода.
- Митигация: ввести годовую подписку «Selenyx Pro» вместо/параллельно с разовыми покупками. Рассмотреть подписку 249₽/мес после набора 100 платящих.

**Риск 2 — Бесплатное слишком щедрое**
Бесплатно доступно: энергия дня, луна, лунный день, совместимость, цвет и число, уведомления. Платный апгрейд сложно продать, если бесплатное уже закрывает 90% потребностей.
- Митигация: переосмыслить разделение до Шага 12. Вынести совместимость или натальную карту за paywall — только тогда тестировать монетизацию.

**Риск 3 — Натальная карта за 690₽ = три шаблонных абзаца**
`SUN_SIGN_DESC` + `MOON_SIGN_DESC` + `ASC_SIGN_DESC` из `data.py`. Пользователь, заплативший ~700 рублей за три строки шаблонного текста, оставит негативный отзыв.
- Митигация: перед запуском монетизации существенно расширить описания, добавить персональный анализ сочетаний (Солнце × Луна × Асцендент как система, не три отдельных поля).

### Контент и retention

**Риск 4 — Контент исчерпывается за 60 дней**
96 комбинаций (12 знаков × 8 фаз) звучат как много. На практике: пользователь со знаком Дева при Растущей Луне видит одинаковый текст 2.5 дня подряд. Через 2 месяца большинство комбинаций пройдено → скука → отток.
- Митигация: добавить третий слой вариативности (день недели, число месяца, ретроград). Или сезонный контент. Рассмотреть GPT-генерацию текстов при базе 500+.

**Риск 5 — Нет нового крючка после первой недели**
Итог недели (streak кратный 7) добавлен, но это один раз в 7 дней. После прохождения онбординга и натальной карты — нет «следующего шага», который ждёт пользователя завтра.
- Митигация: добавить прогрессию (достижения, уровни, визуализация streak). Или еженедельный персональный отчёт с историей прогнозов.

### Технические риски

**Риск 6 — Один процесс, одна точка отказа**
`bot.py` запускает и aiogram-бот, и aiohttp-API. Зависание APScheduler роняет весь сервис включая Mini App.
- Митигация: разделить на два сервиса (в разделе «После MVP»). До этого — добавить healthcheck и авто-рестарт в Railway.

**Риск 7 — SQLite WAL на Railway Volume без мониторинга**
`selenyx.db-shm` и `selenyx.db-wal` — незакоммиченные WAL-файлы. Падение контейнера в момент записи = потеря данных пользователей.
- Митигация: ежедневный бэкап (Fix D из плана митигации) частично решает. Полное решение — миграция на Turso (в «После MVP»).

**Риск 8 — Нет мониторинга ошибок**
Баг в kerykeion при расчёте конкретной даты → бот молча падает для всех. Узнаешь только когда напишут пользователи.
- Митигация: подключить Sentry (бесплатный план, 5000 ошибок/мес). Добавить `/health` алерт на Railway (уже есть эндпоинт, нужен внешний пинг через UptimeRobot).

### Платформенные риски

**Риск 9 — 100% зависимость от Telegram**
Нет email-базы, нет резервного канала связи с пользователями. Любые изменения в политике Telegram Mini App или блокировка = продукт исчезает без возможности уведомить аудиторию.
- Митигация: добавить необязательный сбор email (в форме на лендинге после реального Formspree). Цель: email-список 200+ до монетизации.

**Риск 10 — Telegram Stars недоступны части аудитории**
Stars требуют привязанного способа оплаты в Telegram. Часть аудитории старше 35 никогда не настраивала оплату в Telegram → конверсия ниже ожидаемой.
- Митигация: исследовать альтернативы (ЮКасса через бота, прямые переводы) до Шага 12.

### Привлечение пользователей

**Риск 11 — Лендинг без органического трафика**
Нет SEO-оптимизации, нет ключевых слов в meta, нет блога. Google не будет индексировать осмысленно. Лендинг — просто красивая страница без посетителей.
- Митигация: добавить meta keywords («лунный календарь», «гороскоп на сегодня», «астрология Telegram»). Завести один простой блог-раздел после MVP.

**Риск 12 — Лендинг ведёт на бот, не на Mini App**
Два шага: лендинг → бот → кнопка → Mini App. Прямая ссылка `t.me/Selenyx_mybot/app` открывает Mini App сразу.
- Митигация: заменить все CTA-ссылки на лендинге на прямую deep-link на Mini App. Реализовать до публичного запуска.

### Шаг 7 — риски закрытого теста

**Риск 13 — Знакомые не дадут честной обратной связи**
10–15 приглашённых знакомых будут мягче в оценках. Настоящие инсайты дают незнакомые пользователи из астро-аудитории.
- Митигация: параллельно с личными приглашениями — 1–2 поста в астро-чатах Telegram (анонимно, без просьбы «пожалуйста оцените мягко»).

**Риск 14 — Тест без гипотез = субъективные выводы**
Наблюдение без вопроса не даёт ответа. Нужны конкретные метрики до начала теста.
- Митигация: зафиксировать гипотезы ДО старта теста:
  - Гипотеза A: «30%+ вернутся на день 3» (retention)
  - Гипотеза B: «50%+ откроют натальную карту» (вовлечение)
  - Гипотеза C: «20%+ включат уведомления» (habit formation)

## Конкурентный анализ — что сделал бы конкурент с ресурсами (март 2026)

Взгляд со стороны: что сделал бы конкурент с командой и бюджетом, используя текущие слабые места Selenyx.

### Распределение — главная уязвимость

**PWA вместо Telegram-only**
Selenyx живёт и умирает вместе с Telegram. Умный конкурент строил бы PWA с первого дня: устанавливается на экран, работает как приложение, push-уведомления без Telegram. Telegram — один из каналов, не единственный.
- Что делать: после MVP запустить standalone PWA на отдельном домене. Это страховка и новый канал роста.

**Email-список с первого дня**
При любом сбое Telegram база пользователей исчезает. Конкурент делал бы обязательный email-лид-магнит на лендинге: `"Получи бесплатный лунный прогноз на неделю → на email"`.
- Что делать: подключить Formspree → собирать email через лендинг. Цель: 200+ email до монетизации.

**Deep-link на Mini App, не на бот**
Лендинг → бот → кнопка → Mini App = три шага. Конкурент вёл бы сразу на `t.me/Selenyx_mybot/app`.
- Что делать: заменить все CTA на лендинге до публичного запуска.

### Контент — единственное реальное преимущество

**GPT-генерация уникального контента**
Selenyx показывает одинаковый текст всем Девам при Растущей Луне. Конкурент генерировал бы уникальный текст для каждого пользователя через OpenAI API (~0.02₽ на запрос). Пользователь никогда не встретит «свой» прогноз у подруги — мощный retention.
- Что делать: проработать после набора 500+ пользователей, когда шаблонность станет заметна.

**Контентные «события»**
Selenyx не выделяет особые дни. Конкурент делал бы специальный контент для: лунных затмений, солнечных затмений, входа в ретроград, Полнолуния. Эти события вирусно расходятся сами — люди пересылают «сегодня особенный день».
- Что делать: добавить в `data.py` словарь «особых дат» и специальные тексты для них. Астрологический календарь событий на год.

**Астролог-соавтор как co-creator**
Не консультант — co-creator с долей. Его аудитория (даже 5k подписчиков) даёт старт без рекламного бюджета. Selenyx строит аудиторию с нуля.
- Что делать: найти 1–2 астролога-блогера в Telegram после Шага 7. Предложить коллаборацию: они пишут тексты для одного знака, получают упоминание + % от монетизации пользователей своего знака.

### Виральность — встроенная, не добавленная

**Парный прогноз с петлёй роста**
Не просто совместимость. Пользователь отправляет партнёру специальную ссылку — тот вводит данные, оба получают совместный прогноз на сегодня. Это регистрация нового пользователя через виральную механику.
- Что делать: реализовать в Шаге 10.5+. Ссылка вида `t.me/Selenyx_mybot/app?startapp=pair_TOKEN`.

**Лунный вызов — виральный контент**
`"Твой вызов на Новолуние: начать одно новое дело. Поделись намерением →"` — кнопка шейра встроена в контент. Люди делятся намерениями, это личное и вирально.
- Что делать: добавить «вызов фазы» в Новолуние и Полнолуние с кнопкой поделиться.

**Telegram-канал с ежедневным постом**
Не бот — публичный канал. Каждое утро: общая лунная энергия + ссылка «Твой персональный прогноз →». Канал растёт через репосты, бот монетизирует. Selenyx не имеет публичного присутствия.
- Что делать: создать канал @selenyx после Шага 7 и публиковать ежедневный анонс прогноза.

### Монетизация — принципиально другая логика

**Подписка вместо разовых покупок**
249₽/мес × 1000 пользователей = 249 000₽/мес предсказуемо. Разовые покупки дают потолок 2170₽ с пользователя, после которого платить нечего.
- Что делать: ввести «Selenyx Pro» 249₽/мес параллельно с разовыми покупками. Тестировать оба формата на Шаге 12.

**B2B-вертикаль**
Никто в этом сегменте не думает про бизнес. Конкурент продавал бы HR-командам: «Астрологический профиль команды — совместимость людей в проекте». Один корпоративный клиент = 50 частных пользователей по доходу.
- Что делать: проработать как отдельный продукт после набора 500+ частных пользователей.

### Три самых опасных хода конкурента

| Ход | Почему опасно для Selenyx | Когда закрыть |
|---|---|---|
| GPT-контент + email-лист | Устраняет оба ключевых слабых места разом | После Шага 7 |
| Астролог с аудиторией как co-creator | 5–50k пользователей без рекламного бюджета | Параллельно с Шагом 7 |
| Парный прогноз с виральной петлёй | Встроенный рост, который сложно скопировать | Шаг 10.5+ |

## После MVP — архитектурные улучшения

Задачи, отложенные до набора 100+ активных пользователей и стабилизации продукта:

1. **Разделить bot + API → два Railway-сервиса**
   - Сейчас: один процесс, одно падение = бот + API вниз одновременно
   - Решение: `selenyx-bot` (только aiogram polling) и `selenyx-api` (только aiohttp)
   - Shared DB через Railway Volume с одним writer-процессом

2. **Мигрировать SQLite → Turso** (managed edge SQLite)
   - Сейчас: ежедневный .backup на том же Volume — не защищает от потери Volume
   - Turso: автоматические бэкапы, point-in-time recovery, совместим с aiosqlite через libsql-client-py

3. **Рефакторить monolith webapp/index.html → ES-модули**
   - Сейчас: 2000+ строк инлайн CSS/JS — любое изменение требует контекст всего файла
   - Решение: Vite mini bundle или нативные ES-модули (tab-today.js, tab-moon.js, shared.js)

4. **Тайм-зоны → хранить в users таблице**
   - Сейчас: хардкод Moscow для всего СНГ (Казахстан GMT+5, Беларусь GMT+3)
   - Решение: `timezone TEXT DEFAULT 'Europe/Moscow'` в users + Intl.DateTimeFormat на фронте

5. **Контент-менеджмент → вынести data.py в SQLite/JSON**
   - Сейчас: редактирование контента требует деплоя
   - Решение: таблица `content` (key, value, updated_at) + простой /admin/content endpoint

6. **PWA — standalone версия вне Telegram**
   - Сейчас: 100% Telegram-зависимость, нет резервного канала
   - Решение: PWA на отдельном домене (selenyx.app), push-уведомления через Web Push API

7. **Telegram-канал для органического роста**
   - Сейчас: нет публичного присутствия в Telegram
   - Решение: канал @selenyx, ежедневный пост с общей энергией + deep-link в бот
