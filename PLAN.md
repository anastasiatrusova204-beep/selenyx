# План реализации — Selenyx

> Код пишет Claude. Человек читает, запускает и проверяет результат.
> Каждый шаг = видимый результат. После каждого шага — проверка.

---

## Принципы — чего не повторять за конкурентами

Из отзывов на Saturn.Love, Numia, Co-Star, Moonly, The Pattern — что раздражает пользователей больше всего:

| Что бесит | У кого проблема | Наше правило |
|---|---|---|
| Платить за каждую функцию поверх подписки | Moonly, Co-Star | Платный контент не дублирует бесплатный |
| Много уведомлений | Все | Максимум 1 сообщение в день |
| Гороскоп «для всех» без персонализации | Все | Всегда учитываем знак пользователя |
| Узнать о платном контенте после регистрации | Saturn.Love | Цены видны до оплаты |
| Ответ ждёшь дольше 3 секунд | Все | Бот отвечает мгновенно |

> Нарушение любого из этих правил = потеря пользователя. Это не предположение — это жалобы №1 в реальных отзывах.

## Шаг 6.5 — Минимальная персонализация перед тестом
**Результат:** пользователь видит свой знак в ответе — бот не «забыл» зачем спрашивал.

Что делаем:
- Добавляем в конец «Энергии дня» одну персональную строку по знаку пользователя

- Используем уже сохранённый знак из БД

Проверка: два пользователя с разными знаками видят разные строки в /today.

> **Почему это важно:** тест на шаге 7 должен проверять персонализированный продукт —
> иначе получим обратную связь о том, что мы уже знаем (тексты общие) и ничего нового не узнаем.
> Принцип «Всегда учитываем знак пользователя» должен работать с шага 7, не с шага 9.

## Шаг 7.5 — Нумерология дня

**Результат:** в «Мой день» появляется число дня (1–9) с коротким текстом — контент меняется каждый день, не требует данных от пользователя.

Что делаем:
- Функция `get_day_number(date)` — складывает все цифры даты до однозначного числа (например, 8+3+2026 = 21 → 3)
- Словарь `NUMEROLOGY_DAY` — 9 текстов, по одному на каждое число
- Показываем в «Мой день» новой строкой: `· 🔢 3 день — ...`

**Второй уровень (с натальной картой, Шаг 11):**
- Число судьбы — из даты рождения пользователя
- Число личного года — число судьбы + текущий год
- Персональный совет на основе числа судьбы

Проверка: каждый день в «Мой день» показывается новое число и текст, совпадает с нумерологическим расчётом для текущей даты.

## Шаг 13 — Видимость и продвижение бота

### Поиск внутри Telegram (Telegram SEO)
Telegram ищет ботов только по **username** и **имени бота** — по описанию не ищет.

Что влияет на нахождение в поиске:
- `@username` — сейчас `@Selenyx_mybot`. Если хочется попадать в поиск по «луна» / «астро» — нужен username с ключевым словом (например `@AstroSelenyx`). Менять только если бот ещё не раскручен.
- **Имя бота** (не username) — тоже участвует в поиске. Можно добавить ключевое слово: `Selenyx — лунный навигатор`.
- **Количество пользователей** — чем больше база, тем выше бот в результатах поиска.

Вывод: поиск Telegram даёт малый трафик на старте. Главный канал роста — прямые ссылки.

### Продвижение (реальные каналы трафика)

## Принципы контента — чему учит конкуренция

Эти принципы должны выполняться с первого дня, не когда-то потом:

**1. Психологический фрейм (The Pattern)**
Тексты бота — не «что произойдёт», а «как лучше использовать текущую энергию». Слова «произойдёт», «случится», «ждёт» — вон. Слова «поддерживает», «помогает», «самое время для» — в обиход. Это убирает тревогу и добавляет ощущение контроля.

**2. Читается за 30 секунд (Co-Star, Moonly)**
Максимальная длина одного экрана. Если текст не помещается в одно Telegram-сообщение без скролла — нужно сокращать, а не добавлять.

**3. Контент, который хочется переслать (The Pattern)**
Каждое сообщение должно быть ценным само по себе. Тест: хочется ли скинуть скрин подруге? Если нет — текст нужно переписать.

**4. Растущее качество текстов (Co-Star)**
MVP: авторские шаблоны. При базе 500+ пользователей: GPT-генерация на основе астрологических данных. При монетизации: привлечение астролога-соавтора.
